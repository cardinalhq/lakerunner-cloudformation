# Cardinal cleanup stack — design

## Problem

A Cardinal install consists of two layers an operator cannot tear down with
its own credentials:

1. The **application layer** — the `cardinal-lakerunner` CloudFormation stack
   plus its eight nested children. `cloudformation:DeleteStack` alone is
   normally sufficient, but the lakerunner stack relies on the operator role
   *not* being able to delete customer data (no `s3:DeleteBucket`, no
   `rds:DeleteDBInstance`, no `secretsmanager:DeleteSecret`, no
   `ssm:DeleteParameter`) by design — a compromised Jenkins must not be able
   to wipe a production install.
2. The **data layer** — the resources `scripts/data-setup.sh` provisions
   outside CloudFormation: the `cardinal-ingest-<acct>-<region>` S3 bucket,
   the `cardinal-db` RDS instance (+ subnet group), the `cardinal-ingest` SQS
   queue, the three `cardinal-*` secrets, the two `/cardinal/*` SSM parameters.
   None of these are owned by a CFN stack, so even a `DeleteStack` with
   broader perms wouldn't reach them.

The reference operator-side script `do-not-commit-cleanup.sh` already proves
the pattern: launch a Fargate task in the customer's cluster, assume a
privileged task role, and run a POSIX-sh teardown. But that script is
gitignored, registers the task definition out-of-band each run, and inlines
operator-tweakable shell. It has never shipped.

This spec productizes that pattern as a published, versioned CloudFormation
stack delivered through the same `cardinal-cfn` S3 bucket as the rest of the
template family, plus a committed driver script the operator runs from
Jenkins. **No Lambda anywhere.**

## Hard requirement

The cleanup task **only touches resources Cardinal created**: the
`cardinal-lakerunner` stack and everything it owns; the data resources
`data-setup.sh` provisioned (S3 ingest bucket, RDS instance, RDS subnet
group, SQS queue, the three `cardinal-*` secrets, the two `/cardinal/*` SSM
parameters); and the `cardinal-cleanup` stack itself.

It never touches customer-owned ("BYO") inputs: the ECS cluster, the Cloud
Map namespace, the task role, the execution role, the deployer role, any
security group, any subnet, the VPC, or the optional `cardinal-vpc` stack.
These appear in the design only as read-only inputs.

### Ownership-tag enforcement (defense in depth)

Cardinal's deterministic resource names (`cardinal-db`, `cardinal-ingest-*`,
`cardinal-db-master`, etc.) are not by themselves proof of ownership — a
customer might happen to have a pre-existing resource with that name, or a
prior aborted install. **Before issuing any destructive call against a
data-layer resource, the cleanup script reads the resource's tags and
refuses to delete unless the ownership tags are present.**

`scripts/data-setup.sh:71-73` already tags every resource it creates with:

```
Application = cardinal-lakerunner
Project     = cardinal
ManagedBy   = cardinal-data-setup-script
Component   = <per-resource>
Name        = cardinal-<component>
```

The cleanup script's ownership predicate is:

```
ManagedBy == "cardinal-data-setup-script"  AND  Application == "cardinal-lakerunner"
```

A resource that lacks both tags or carries different values is **logged
loudly and skipped** — never deleted. The script's exit code surfaces these
skips as a non-zero (`exit 1`) so the operator sees the partial cleanup in
Jenkins. Tag fetch is via the resource's own API (e.g.
`secretsmanager:DescribeSecret`, `s3:GetBucketTagging`,
`rds:ListTagsForResource`, `sqs:ListQueueTags`, `ssm:ListTagsForResource`)
not Resource Groups Tagging API, to avoid an extra IAM verb.

### Drain restricts to stack resources, not cluster contents

The customer's ECS cluster may hold other services unrelated to Cardinal.
The drain step therefore derives its target list from
`cloudformation:ListStackResources` on the `cardinal-lakerunner` root +
nested children — never from `ecs:ListServices` on the cluster — so the
cleanup cannot touch a non-Cardinal service that happens to share the
cluster. Since the lakerunner generators do not set `ServiceName`
(`src/cardinal_cfn/children/services_query.py`,
`services_process.py`, `services_control.py`), every service's physical id
is a CFN-auto-generated name like
`cardinal-lakerunner-ServicesQuery-QueryApi-A1B2C3D4-...`; the drain reads
that physical id from `ListStackResources` and uses it directly.

## Approach

Add a new sibling root template `cardinal-cleanup.yaml`, generated by
`src/cardinal_cfn/cardinal_cleanup.py` and published to the same S3 prefix
as `cardinal-lakerunner.yaml`. The template owns exactly two resources:

- `AWS::Logs::LogGroup` — explicit name `/aws/ecs/cardinal-cleanup/${AWS::StackName}`,
  7-day retention, `DeletionPolicy: Delete`.
- `AWS::ECS::TaskDefinition` — Fargate, `512 CPU / 1024 MiB`, network mode
  `awsvpc`, single container running `public.ecr.aws/aws-cli/aws-cli:latest`
  with an inline POSIX-sh teardown as the `command`. `TaskRoleArn` and
  `ExecutionRoleArn` come from parameters (customer-supplied — same shape
  as the lakerunner stack). The log group is referenced via `LogConfiguration`.

The stack carries no IAM, no security group, no networking, no
`AWS::ECS::Service`. It is a vehicle to deliver a vetted, S3-distributed task
definition that the operator can launch but cannot tamper with.

### Why the script lives in `EntryPoint`, not `Command`

`ecs:RunTask`'s `containerOverrides` lets the caller override the
container's `command`, `environment`, `cpu`, and `memory` — but **not its
`entryPoint`**. Putting the inline script in `EntryPoint` means an operator
with `ecs:RunTask` cannot substitute a different command (e.g.
`["sh", "-c", "aws s3 rm ..."]`) to bypass the vetted script while still
running as the privileged task role. Together with the
`cloudformation:TemplateUrl` condition on `CreateStack`, this closes the
remaining tamper vector: the operator can neither change what the template
provisions nor change what the launched task actually runs.

Environment overrides remain possible via `containerOverrides.environment`,
including `AWS_REGION` / `AWS_DEFAULT_REGION` (which the AWS CLI honors).
A naive design with wildcarded ARNs would let a compromised Jenkins
re-target a Cardinal install in *another region of the same account* just
by overriding `AWS_REGION` at run-task time. The defense is **scoping IAM
to a concrete account + region + cluster + bucket** rather than relying
on the script's env values:

- The customer-supplied task role is created in a specific account, and
  its IAM policy uses **the exact account number and region in every
  ARN** — `arn:aws:s3:::cardinal-ingest-<ACCOUNT>-<REGION>` (not `*`),
  `arn:aws:rds:<REGION>:<ACCOUNT>:db:cardinal-db` (not `*:*:*`),
  `arn:aws:ecs:<REGION>:<ACCOUNT>:service/<CLUSTER-ARN>/*`, etc. An
  `AWS_REGION` override sends the API call to the wrong region; the IAM
  ARNs don't match; the call 403s.
- The script derives the account via `aws sts get-caller-identity` (which
  uses task-role credentials, not env) — never from `AWS_ACCOUNT_ID` env.
  Region is read from the ECS task metadata
  (`${ECS_CONTAINER_METADATA_URI_V4}/task` -> `TaskARN` -> parse region
  segment) — never from `AWS_REGION` env. If those derivations and the
  env values disagree, the script aborts with a non-zero exit before
  issuing any destructive call.
- The operator's `ecs:RunTask` policy adds two conditions:
  `ecs:cluster = <exact cluster ARN>` (so RunTask is bounded to the
  intended cluster) and `StringEquals: { ecs:enable-execute-command: "false" }`
  (so an operator with `ecs:ExecuteCommand` rights elsewhere cannot
  open an interactive shell inside the privileged task by toggling
  exec on at run-task time).
- The cleanup task definition omits `EnableExecuteCommand` (default
  `false`) and the cluster's `executeCommandConfiguration` is irrelevant
  because the operator condition forbids enabling it at all.

Together these mean that overriding env vars doesn't grant the operator a
re-target capability: every API call's IAM check independently validates
account, region, cluster, and resource name against the role's literal
ARN — not against any env-supplied value.

A new committed driver `scripts/cleanup-lakerunner.sh` orchestrates the run:

```text
Operator (Jenkins)           cardinal-cleanup stack         ECS Fargate task
─────────────────────────    ──────────────────────────    ──────────────────────────
cleanup-lakerunner.sh        cardinal-cleanup.yaml         public.ecr.aws/aws-cli
  create-stack ──────────►   TaskDefinition + LogGroup
  run-task ─────────────►                          ──────► /bin/sh -c <inline-script>
  tail logs ◄─────────                                       1. enumerate cardinal-lakerunner
  exit(task-code) ◄───                                          ECS services via CFN; set
                                                                desired=0; stop-task each;
                                                                wait for drain.
                                                              2. delete cardinal-lakerunner
                                                                 CFN stack (+ wait).
                                                              3. for each data resource:
                                                                 verify ownership tags
                                                                 (Application=cardinal-
                                                                 lakerunner, ManagedBy=
                                                                 cardinal-data-setup-
                                                                 script); skip + warn if
                                                                 mismatch; else: drain +
                                                                 delete S3 ingest (incl.
                                                                 abort multipart uploads);
                                                                 modify+delete RDS; delete
                                                                 subnet group; delete SQS;
                                                                 force-delete the three
                                                                 cardinal-* secrets; delete
                                                                 the two /cardinal/* SSM
                                                                 parameters.
                             (deleted by task) ◄─────────── 4. fire delete-stack
                                                                cardinal-cleanup (async,
                                                                no wait).
                                                              5. exit 0 if every step
                                                                 succeeded; exit 1 if any
                                                                 ownership skip or delete
                                                                 failure occurred.
```

The "task does everything" split exists because the operator role is
deliberately narrow. Once the cleanup stack has been created, all the
destructive verbs are held by the task role. The driver's only role-level
permissions are `cloudformation:CreateStack/DescribeStacks/DeleteStack`
on `cardinal-cleanup`, `ecs:RunTask` + `iam:PassRole` on the two
customer-supplied role ARNs, and `logs:GetLogEvents` for tailing.

Self-delete works because `cloudformation:DeleteStack` is fire-and-forget —
the task issues the call, then exits. CFN proceeds with the actual teardown
asynchronously. The only resources to tear down are the log group (named
with the stack-name suffix so each cleanup run gets a fresh group) and the
task definition (CFN deregisters
it; running tasks are unaffected by deregistration). The customer-supplied
task and execution roles are not owned by the stack, so they survive.

## `cardinal-cleanup.yaml` — parameters, resources, outputs

### Parameters

| Name | Type | Required | Notes |
|---|---|---|---|
| `LakerunnerStackName` | String | yes | Default `cardinal-lakerunner`. Allows non-default install names. |
| `CleanupTaskRoleArn` | String | yes | Privileged IAM role; trust policy must allow `ecs-tasks.amazonaws.com`. Holds all destructive verbs (see below). |
| `CleanupExecutionRoleArn` | String | yes | Standard Fargate execution role (ECR pull + log writes). May be the same role as `CleanupTaskRoleArn` if customer has consolidated. |
| `ClusterName` | String | yes | Customer's ECS cluster (the same one lakerunner services run in). Baked into the task as `CLUSTER_NAME` env. |
| `DeployerRoleArn` | String | yes | Same `cardinal-cfn-deployer` role used by `deploy-lakerunner.sh`. Baked into the task as `DEPLOYER_ROLE_ARN` env so the in-task `delete-stack` calls pass `--role-arn`. |

The container image is **hard-coded in the template** as
`public.ecr.aws/aws-cli/aws-cli:latest` and is intentionally NOT a parameter.
A parameter would let an operator with `cloudformation:CreateStack` on
`cardinal-cleanup` point the privileged task role at a malicious image,
defeating the `cloudformation:TemplateUrl` tamper guard. Air-gapped
customers without NAT egress mirror this image *and* the template into
their own infrastructure, then deploy from the mirrored template URL
(which their operator IAM `TemplateUrl` condition allows) — they do not
override the image at deploy time.

No `NoEcho` parameters. No secrets in this template.

## Inline shell script — sequence and semantics

Lifted from `do-not-commit-cleanup.sh:84-191` with a new pre-step (service
drain) and a new tail step (self-delete). Runs on
`public.ecr.aws/aws-cli/aws-cli:latest`, which has `aws`, `python3`, `sh`,
but **no `jq`** — all JSON munging goes through inline `python3 -c '...'`.

**Pre-step (security):** Before doing any work, the script derives the
account and region from authoritative sources (not env vars an operator
could override):

- `ACCOUNT=$(aws sts get-caller-identity --query Account --output text)`
  — returns the account the task-role credentials live in; cannot be
  influenced by `AWS_ACCOUNT_ID` env.
- Region is parsed from the task ARN reachable via the ECS task metadata
  endpoint, fetched via `python3` (the container image guarantees
  `python3` is present; `curl` is not in the dependency contract):
  `python3 -c "import os,urllib.request,json; uri=os.environ['ECS_CONTAINER_METADATA_URI_V4']; print(json.load(urllib.request.urlopen(uri+'/task'))['TaskARN'].split(':')[3])"`.

  Note that `ECS_CONTAINER_METADATA_URI_V4` is itself an env var, and is
  in principle overridable via `containerOverrides.environment`. This
  defense is therefore **defense-in-depth, not the primary control** —
  the primary control against cross-account / cross-region redirect is
  the account+region pinning of every destructive ARN in the task-role
  IAM policy. The metadata-derived region only catches an attacker who
  somehow has matching IAM but supplies a misleading `AWS_REGION` env
  to confuse the CLI's default-region resolution; in practice IAM
  already denies first.

Then the script asserts that `LAKERUNNER_STACK_NAME` and
`CLEANUP_STACK_NAME` are non-empty and that the derived account/region
are non-empty. The script does **not** treat the env values themselves as
trusted (they are overridable via `ecs:RunTask` containerOverrides); the
bound on env tampering is IAM scope, not env validation. Specifically:

- The single-install assumption (CLAUDE.md: one Cardinal install per
  account/region) means an attacker cannot redirect at a sibling install
  in the same region — there isn't one.
- Cross-region redirect is blocked by the metadata-derived region:
  `AWS_DEFAULT_REGION` is unconditionally re-exported from the parsed
  task ARN, so all CLI calls hit the actual region the task is running
  in. Customer task-role ARNs are region-pinned (see IAM section), so a
  cross-region API call hits a `Deny` even if the env said otherwise.
- Cross-account redirect is blocked by the STS-derived account: the
  task-role credentials only resolve in the account they were created
  in. `s3:DeleteBucket arn:aws:s3:::cardinal-ingest-<other-acct>-...`
  fails because the credentials cannot speak for that account.

Steps (every step is idempotent; "already absent" is treated as success;
every data-layer step is ownership-tag-gated):

1. **Drain Cardinal ECS services.** Recursively walk `cardinal-lakerunner`
   root + nested children via `aws cloudformation list-stack-resources`
   (paginated via `NextToken`; filter to `AWS::CloudFormation::Stack` and
   `AWS::ECS::Service`). For each `AWS::ECS::Service`, the
   `PhysicalResourceId` is the full service ARN
   (`arn:aws:ecs:<region>:<acct>:service/<cluster>/<name>`); normalize to
   the service NAME (last `/`-segment) for `ecs:UpdateService --service`
   and `ecs:ListTasks --service-name`, since those CLI flags take names,
   not ARNs. Use the full ARN where AWS does accept ARNs
   (`ecs:DescribeServices --services`).

   For each service: `aws ecs update-service --cluster $CLUSTER_NAME
   --service <name> --desired-count 0`. Call `aws ecs list-tasks
   --cluster $CLUSTER_NAME --service-name <name>` twice (once with
   `--desired-status RUNNING` and once with `--desired-status PENDING`),
   union the results, and `aws ecs stop-task --cluster $CLUSTER_NAME
   --task <id> --reason "cardinal-cleanup drain"` each.

   Poll `aws ecs describe-services --cluster $CLUSTER_NAME --services
   <ARNs in batches of 10>` (the API caps each call at 10) until every
   service reports `runningCount=0 AND pendingCount=0`, with a 5-minute
   cap. After the cap, log a warning and proceed — `delete-stack` will
   force-stop any stragglers.
2. **Delete `cardinal-lakerunner`.** `aws cloudformation delete-stack
   --stack-name $LAKERUNNER_STACK_NAME --role-arn $DEPLOYER_ROLE_ARN`,
   then `wait stack-delete-complete`. The task always passes the
   deployer role (never the role recorded on the stack, even if one is
   present), so the task role's IAM only needs `iam:PassRole` on the
   single, well-known deployer-role ARN — not on whatever role the stack
   happened to be created with. Treat `DOES_NOT_EXIST` as success. The
   stack delete can take 10+ minutes (ALB drain, ECS service deletion,
   target group cleanup).
3. **Drain + delete S3 ingest bucket** `cardinal-ingest-${ACCOUNT}-${REGION}`.
   Verify ownership tags first (`s3:GetBucketTagging`) — abort that step
   if `ManagedBy/Application` are not the Cardinal values. Then:
   `aws s3 rm s3://BUCKET --recursive` for current objects; loop
   `list-object-versions` + `delete-objects` for versioned buckets;
   `aws s3api list-multipart-uploads` + `abort-multipart-upload` for any
   in-flight uploads (otherwise `delete-bucket` errors `BucketNotEmpty`);
   finally `delete-bucket`.
4. **Delete RDS** `cardinal-db`: `describe-db-instances` (also surfaces
   `TagList`); verify ownership tags; if `DeletionProtection=True`,
   `modify-db-instance --no-deletion-protection --apply-immediately`;
   `delete-db-instance --skip-final-snapshot --delete-automated-backups`;
   `wait db-instance-deleted`.
5. **Delete RDS subnet group** `cardinal-db-subnet-group`. Verify
   ownership tags via `rds:ListTagsForResource` against the subnet group
   ARN before delete.
6. **Delete SQS queue** `cardinal-ingest`. Resolve queue URL via
   `get-queue-url`; verify ownership tags via `sqs:ListQueueTags` against
   the URL before delete.
7. **Force-delete the three `cardinal-*` secrets** (`cardinal-db-master`,
   `cardinal-license`, `cardinal-admin-key`) with
   `--force-delete-without-recovery`. Verify ownership tags via
   `secretsmanager:DescribeSecret` before each delete.
8. **Delete the two SSM parameters** `/cardinal/storage-profiles`,
   `/cardinal/api-keys`. Verify ownership tags via
   `ssm:ListTagsForResource --resource-type Parameter --resource-id <name>`
   before each delete.
9. **Self-delete.** `aws cloudformation delete-stack --stack-name
   $CLEANUP_STACK_NAME --role-arn $DEPLOYER_ROLE_ARN`. The deployer role
   passed by the driver is what CFN uses to actually remove the cleanup
   stack's task definition and log group — the customer-supplied task
   role does NOT have `ecs:DeregisterTaskDefinition` or
   `logs:DeleteLogGroup`, and shouldn't. No wait — CFN runs async; the
   task exits before CFN gets to the log group / task-def teardown.
   The container's last log lines may be dropped when the log group is
   deleted; this is acceptable (driver has already observed task `STOPPED`
   and exit code by then). If the self-delete API call itself errors, the
   task logs it and still exits 0 — the destructive work is already done;
   the operator can delete the now-empty cleanup stack manually.
10. **`exit 0`** if every prior step succeeded. **`exit 1`** if any
    ownership tag mismatch caused a skip, any tagged resource delete
    failed, or any presence-checked resource delete failed. Step 9 (self-
    delete) does not affect the exit code.

The script tolerates re-runs against a partially-cleaned account: each step
checks for resource presence before issuing any destructive call, and absent
resources are logged and skipped. A second cleanup invocation against a
fully-cleaned account is therefore a no-op that exits 0. An ownership-tag
mismatch on any resource fails closed (skip + non-zero exit).

## CFN service role threading (`--deployer-role-arn` is required)

Both the cleanup stack's create-and-delete lifecycle and the inline script's
deletes of the lakerunner stack and the cleanup stack itself depend on
CloudFormation acting through a service role with broad enough permissions
to manage the underlying resources (the ECS task definition, the log group,
and everything the lakerunner stack owns). The operator role intentionally
lacks these verbs (`ecs:RegisterTaskDefinition`, `ecs:DeregisterTaskDefinition`,
`logs:CreateLogGroup`, `logs:DeleteLogGroup`, plus the entire lakerunner
delete surface). The customer-supplied task role intentionally lacks the
ECS task-definition + log-group verbs too — it has destructive data-layer
perms, not stack-mechanics perms.

**Resolution:** `--deployer-role-arn` is **required** on the driver
(promoted from optional). It is the same `cardinal-cfn-deployer` role
documented in `docs/operations/deploying.md` and used by `deploy-lakerunner.sh`.
The driver:

1. Passes `--role-arn $DEPLOYER_ROLE_ARN` to `cloudformation:CreateStack`
   when creating the cleanup stack (so CFN provisions the task def + log
   group as the deployer, not as Jenkins).
2. Passes `$DEPLOYER_ROLE_ARN` to the task as an env var
   (`DEPLOYER_ROLE_ARN`), which the inline script then uses for both the
   `delete-stack cardinal-lakerunner --role-arn ...` call and the
   `delete-stack cardinal-cleanup --role-arn ...` self-delete call.

The cleanup stack's task role then needs only `iam:PassRole` on the
deployer role (so the in-task `aws cloudformation delete-stack --role-arn`
calls succeed) and `cloudformation:DeleteStack` on the two stack ARNs.
It does NOT need any ECS task-definition or log-group verb directly.

## Customer-supplied `CleanupTaskRoleArn` — required policy

Documented in `docs/operations/cleanup.md`. Trust policy must allow
`ecs-tasks.amazonaws.com:AssumeRole`. Required permissions:

```text
# CloudFormation discovery (drain) and stack deletes. Nested children get
# auto-generated stack names of the form `cardinal-lakerunner-<Child>Stack-<HASH>`,
# so the resource ARN scope must cover both the root and the children. All
# ARNs are account+region pinned.
cloudformation:ListStackResources       arn:aws:cloudformation:<REGION>:<ACCOUNT>:stack/cardinal-lakerunner/*
                                        arn:aws:cloudformation:<REGION>:<ACCOUNT>:stack/cardinal-lakerunner-*/*
cloudformation:DescribeStacks           same two patterns plus
                                        arn:aws:cloudformation:<REGION>:<ACCOUNT>:stack/cardinal-cleanup/*
cloudformation:DescribeStackResources   same as ListStackResources
cloudformation:DeleteStack              arn:aws:cloudformation:<REGION>:<ACCOUNT>:stack/cardinal-lakerunner/*
                                        arn:aws:cloudformation:<REGION>:<ACCOUNT>:stack/cardinal-cleanup/*

# Required: PassRole on the deployer role so the in-task `delete-stack --role-arn`
# calls can hand it to CloudFormation. Without this, both the lakerunner-stack
# delete (step 2) and the cleanup-stack self-delete (step 9) fail.
iam:PassRole                            <DeployerRoleArn>
  Condition: { StringEquals: { "iam:PassedToService": "cloudformation.amazonaws.com" } }

# ECS drain. Lakerunner does not set ServiceName, so service ARNs are
# CFN-auto-generated. ECS services in nested children carry the auto-applied
# `aws:cloudformation:stack-name` tag set to the CHILD stack's name (e.g.
# `cardinal-lakerunner-ServicesQueryStack-A1B2C3D4`), not the root name —
# so the IAM condition uses StringLike with the root prefix to match all
# children. On calls that do not support the resource-tag condition
# (ListTasks, StopTask, DescribeTasks), scope to the cluster path and rely
# on the script working only from ARNs it discovered via CFN.
ecs:UpdateService                       arn:aws:ecs:<REGION>:<ACCOUNT>:service/<CLUSTER>/*
  Condition: { StringLike: { "aws:ResourceTag/aws:cloudformation:stack-name": "cardinal-lakerunner*" } }
ecs:DescribeServices                    arn:aws:ecs:<REGION>:<ACCOUNT>:service/<CLUSTER>/*
ecs:ListTasks                           *  (AWS does not allow resource-level scoping for this action)
  Condition: { ArnEquals: { "ecs:cluster": "arn:aws:ecs:<REGION>:<ACCOUNT>:cluster/<CLUSTER>" } }
```

## Operator role — required policy

The operator (Jenkins) role needs:

```text
cloudformation:CreateStack          arn:aws:cloudformation:*:*:stack/cardinal-cleanup/*
  Condition: {
    StringEquals: { "cloudformation:RoleArn": "<DeployerRoleArn>" },
    StringLike:    { "cloudformation:TemplateUrl": [
      "https://cardinal-cfn.s3.us-east-2.amazonaws.com/lakerunner/*/cardinal-cleanup.yaml",
      "https://cardinal-cfn-us-east-1.s3.us-east-1.amazonaws.com/lakerunner/*/cardinal-cleanup.yaml"
      # Air-gapped customers add their exact mirror URL here. Wildcarding the
      # bucket host is forbidden — any bucket matching `cardinal-cfn-*` could be
      # created by anyone and would defeat the tamper guard.
    ] }
  }
cloudformation:DescribeStacks       arn:aws:cloudformation:*:*:stack/cardinal-cleanup/*
cloudformation:DescribeStackEvents  same
cloudformation:DeleteStack          same  (for stranded-stack recovery only;
                                           normal runs delete via the task)
  Condition: { StringEquals: { "cloudformation:RoleArn": "<DeployerRoleArn>" } }

ecs:RunTask                         arn:aws:ecs:<REGION>:<ACCOUNT>:task-definition/cardinal-cleanup:*
  Condition: {
    ArnEquals:    { "ecs:cluster": "arn:aws:ecs:<REGION>:<ACCOUNT>:cluster/<CLUSTER>" },
    StringEquals: { "ecs:enable-execute-command": "false" }
  }
ecs:DescribeTasks                   arn:aws:ecs:<REGION>:<ACCOUNT>:task/<CLUSTER>/*
  Condition: { ArnEquals: { "ecs:cluster": "arn:aws:ecs:<REGION>:<ACCOUNT>:cluster/<CLUSTER>" } }

iam:PassRole                        <CleanupTaskRoleArn>
                                    <CleanupExecutionRoleArn>
  Condition: { StringEquals: { "iam:PassedToService": "ecs-tasks.amazonaws.com" } }

iam:PassRole                        <DeployerRoleArn>
  Condition: { StringEquals: { "iam:PassedToService": "cloudformation.amazonaws.com" } }

logs:GetLogEvents                   arn:aws:logs:*:*:log-group:/aws/ecs/cardinal-cleanup/*:log-stream:*
logs:DescribeLogStreams             arn:aws:logs:*:*:log-group:/aws/ecs/cardinal-cleanup/*
```

The `cloudformation:TemplateUrl` condition on `CreateStack` is the mechanism
that prevents an operator from substituting an arbitrary task definition
under the `cardinal-cleanup` stack name. Without it, an operator with
`ecs:RunTask` + `iam:PassRole` on the cleanup task role could deploy any
template and run any command against the priv'd role. **This condition is
load-bearing and must be documented as required in the operator runbook.**
Air-gapped customers using `--template-base-url` to point at a mirror must
update the condition's URL pattern accordingly.

This is the same shape as the existing `cardinal-cfn-deployer` policy plus
`ecs:RunTask` + `iam:PassRole`. It explicitly does **not** grant any data-
layer verb.

## `scripts/cleanup-lakerunner.sh` — driver

POSIX shell, AWS CLI v2 + jq, self-contained (no Python package dependency).
Matches the conventions of `scripts/deploy-lakerunner.sh` and
`scripts/teardown-lakerunner.sh`: `--flag value` style; `internal_*` test
hooks for pure data transforms; explicit exit-code contract.

### CLI surface

```text
Usage: cleanup-lakerunner.sh [options]

Required:
  --region REGION                       AWS region.
  --version VERSION                     Published template tag, e.g. v0.0.46.
  --cluster-name NAME                   Customer's ECS cluster.
  --private-subnets CSV                 Subnets for the cleanup task ENI.
  --task-sg-id SG_ID                    Security group for the cleanup task ENI.
  --cleanup-task-role-arn ARN           Priv'd task role.
  --cleanup-execution-role-arn ARN      Fargate execution role.
  --deployer-role-arn ARN               CFN service role (cardinal-cfn-deployer). REQUIRED.
                                        Used for create-stack on cardinal-cleanup, and threaded
                                        into the task as $DEPLOYER_ROLE_ARN for the in-task
                                        delete-stack --role-arn calls.
  --yes                                 Confirms destructive operation.

Optional (defaults are the documented Cardinal stack names; non-default
values require the operator-role IAM policy ARN scopes to be rewritten):
  --lakerunner-stack-name NAME          Default: cardinal-lakerunner.
  --cleanup-stack-name NAME             Default: cardinal-cleanup.
  --template-base-url URL               Default: https://cardinal-cfn-us-east-1.s3.us-east-1.amazonaws.com/lakerunner.
  --wait-self-delete                    Wait for cardinal-cleanup stack-delete-complete after task exit
                                        (off by default per the chosen seam shape).
```

### Flow

1. Parse + validate args. Missing `--yes` → print plan, exit 2.
2. Print the destructive blast-radius (target account, region, stack names,
   cluster) and require `--yes`.
3. If `cardinal-cleanup` already exists in any non-deleted state,
   `aws cloudformation delete-stack --stack-name cardinal-cleanup
   --role-arn $DEPLOYER_ROLE_ARN` and `wait stack-delete-complete` first.
   Stranded stacks come from prior aborted runs. Passing `--role-arn` is
   required because the operator's `cloudformation:DeleteStack` policy has
   a `cloudformation:RoleArn` StringEquals condition.
4. `aws cloudformation create-stack --stack-name cardinal-cleanup
   --template-url <S3>/v.../cardinal-cleanup.yaml --capabilities CAPABILITY_IAM
   --role-arn $DEPLOYER_ROLE_ARN` with the parameters. Wait
   `stack-create-complete`. `--role-arn` is required (operator IAM
   condition); CFN provisions the task def + log group as the deployer
   role, not as Jenkins.
5. Read `TaskDefinitionArn` and `LogGroupName` from stack outputs.
6. `aws ecs run-task --cluster $CLUSTER_NAME --launch-type FARGATE
   --task-definition $TD_ARN --network-configuration
   awsvpcConfiguration={subnets=[...],securityGroups=[...],assignPublicIp=DISABLED}`.
7. Poll `describe-tasks` until `lastStatus=RUNNING`; fail with full
   `stopCode` / `stoppedReason` / container exit codes if it stops first.
8. Tail CloudWatch Logs from `cleanup/cleanup/<TASK_ID>` until task
   `lastStatus=STOPPED`. Print each event line as it arrives.
9. Read `containers[0].exitCode`.
10. (Default off) If `--wait-self-delete`, `aws cloudformation wait
    stack-delete-complete --stack-name cardinal-cleanup` with a polling cap.
11. `exit` with the task exit code (or 1 if step 10 timed out).

### Exit codes

- `0` — task succeeded; cleanup stack delete in progress (and confirmed
  complete if `--wait-self-delete`).
- `1` — task ran and failed (non-zero exit), self-delete failed/timed out,
  or any AWS call errored unexpectedly.
- `2` — pre-flight failure (missing required flag, missing `--yes`,
  malformed input).

## Error handling

- **Cleanup stack already exists.** Driver pre-deletes, then proceeds.
- **Lakerunner stack absent.** Inline script step 2 treats `DOES_NOT_EXIST`
  as success (idempotent re-runs).
- **Data resources absent.** Each step in the inline script presence-checks
  before issuing a delete. Already-absent resources log and are skipped.
- **RDS deletion-protection on.** Script disables it before delete.
- **Versioned S3 bucket.** Script clears versions + delete markers in a loop
  before `delete-bucket`.
- **Task stops during startup** (image pull failure, ENI quota, etc.).
  Driver fails immediately with full `stopCode` + container exit details from
  `describe-tasks`. Cleanup stack remains; operator deletes it manually or
  re-runs (which pre-deletes).
- **Task runs but exits non-zero.** Driver tails logs to completion, returns
  the container exit code as its own. Cleanup stack remains for inspection
  (the task fires self-delete only after a clean run).
- **Self-delete `delete-stack` call inside the task fails.** Logged; the
  task still exits 0 because the destructive work is already done. Operator
  re-runs with `--yes` to re-create-then-delete, or deletes manually. The
  driver only surfaces a self-delete failure if `--wait-self-delete` was
  passed; in that mode, the driver exits 1 after the timeout.
- **Ownership-tag mismatch on any data-layer resource.** That resource is
  logged loudly and skipped; the script continues with subsequent steps;
  the final exit code is 1. The cleanup stack is **not** self-deleted in
  this case — the operator inspects the log, decides whether the surviving
  resource is theirs or an artifact, and re-runs (after manual fixup) or
  deletes the cleanup stack manually.
- **Drain step timeout.** After 5 minutes of services not reaching
  `runningCount=0`, script logs a warning and proceeds — `delete-stack`
  will force-stop them.

## VPC endpoint / egress requirements

The cleanup task runs in the customer-supplied private subnets with no
public IP. The container image is `public.ecr.aws/aws-cli/aws-cli:latest`,
and **public-ECR image pulls from private subnets require either a NAT
gateway or a customer mirror of the image into private ECR** — the
standard `ecr-api` / `ecr-dkr` VPC interface endpoints do not serve
`public.ecr.aws`. NAT is the simplest path; if the customer's posture
forbids NAT, see the airgapped-mirror note below — the cleanup template
itself must be rebuilt with the mirrored image URI, because the image is
intentionally not a stack parameter.

Once the image is reachable, the task must additionally reach these AWS
API endpoints, either via NAT or via VPC interface endpoints
(`com.amazonaws.<region>.<service>`):

- `cloudformation` (drain + delete-stack)
- `ecs` (drain + self-introspection)
- `s3` (object/version/multipart delete, bucket delete; gateway endpoint suffices)
- `rds` (modify + delete + describe + tag fetch)
- `sqs` (delete + tag fetch)
- `secretsmanager` (describe + delete + tag fetch)
- `ssm` (delete + tag fetch)
- `logs` (write stream)
- `sts` (caller identity)

The cleanup template's container image is hard-coded
(`public.ecr.aws/aws-cli/aws-cli:latest`) — see the template-parameters
section for why a parameter would re-open a tamper vector. Air-gapped
customers without NAT must mirror both the image and the cardinal-cleanup
template into their own infrastructure (rebuilding the template with the
mirrored image URI) and point the operator role's `TemplateUrl` condition
at the mirrored template.

The operator runbook calls out the full endpoint list and the
mirror-or-NAT decision.

## Release

`build.sh` and the Makefile lint target both enumerate generated templates
explicitly. **Two edits are required:**

- `build.sh:21` (and the cfn-lint invocation at the bottom): add a step
  generating `cardinal-cleanup.yaml` from `cardinal_cfn.cardinal_cleanup`.
- `Makefile:37-43` (`lint` target): add
  `generated-templates/cardinal-cleanup.yaml` to the cfn-lint argument list.

`.github/workflows/release.yml` publishes everything under
`generated-templates/` to S3 under the version prefix; once the file lands
in that directory, the release wires it up automatically.

The driver script `scripts/cleanup-lakerunner.sh` ships via the repo
(committed), not via S3, mirroring `deploy-lakerunner.sh` and
`teardown-lakerunner.sh`.

## Out of scope

- Deleting the optional `cardinal-vpc` stack. The customer does not use it
  in the target deployments; if they did, it would be deleted out-of-band
  by the customer team using their own creds.
- Cleaning up customer-owned IAM roles, security groups, VPC, subnets, ECS
  cluster, Cloud Map namespace.
- Cleaning up unrelated tenants of the customer's ECS cluster.
- Bringing the cleanup task definition under autoscaling, image-pinning,
  or per-region replication — it's a one-shot, public-image teardown.
- Replacing or deprecating `scripts/teardown-lakerunner.sh` (which handles
  legacy pre-pivot installs); the two scripts coexist with different
  target install shapes.
